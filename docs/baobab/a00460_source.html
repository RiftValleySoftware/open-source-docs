5 COBRA
6 ======
7 
8 Part of the BAOBAB Server, which is part of the Rift Valley Platform
9 --------------------------------------------------------------------
10 ![BAOBAB Server and The Rift Valley Platform](images/BothLogos.png)
11 
12 INTRODUCTION
13 ============
14 COBRA is a security administration toolset for the Rift Valley Platform (RVP).
15 
16 ![COBRA](images/COBRALayers.png)
17 
18 It is a standalone class that is instantiated with a logged-in \ref CHAMELEON instance. That login needs to be an instance of `CO_Login_Manager` (or a subclass).
19 
20 COBRA will allow a manager to create and delete logins, as well as manage security tokens.
21 
22 \ref CHAMELEON is the "First Layer Abstraction" from the data storage and retrieval. It implements a few higher-level capabilities, such as collections, users and places.
23 
24 \ref BADGER is the "First Layer Connection" to the data storage subsystem. It uses [PHP PDO](http://php.net/manual/en/book.pdo.php) to abstract from the databases, and provide SQL-injection protection through the use of [PHP PDO Prepared Statements](http://php.net/manual/en/pdo.prepared-statements.php).
25 
26 DESCRIPTION
27 ===========
28 
29 COBRA is designed to be the "user manager" for The Rift Valley Platform. It is not required to run the RVP, and is not required for basic logins and control of general security access. That is handled by \ref BADGER and \ref CHAMELEON.
30 
31 COBRA is required to create and manage logins and users.
32 
33 NEW LOGIN CLASSES
34 -----------------
35 
36 COBRA introduces `CO_Cobra_Login` and `CO_Login_Manager`, two specializations of `CO_Security_Login` (from \ref BADGER). These are available whether or not an instance of CO_Cobra is available.
37 
38 `CO_Cobra_Login` describes your "standard" endpoint login.
39 
40 `CO_Login_Manager` is a class for "managers." These are logins that can create and delete other logins.
41 
42 If a `CO_Login_Manager` (or "God") deletes a login, the login is not actually removed from the database. Instead, it is converted to a security token.
43 
44 TOKEN MANAGER
45 -------------
46 
47 A `CO_Login_Manager` can use COBRA to create new security tokens (`CO_Security_ID` from \ref BADGER). These are security database rows that have no purpose, other than to fill a slot.
48 
49 Once created, you cannot delete a security token.
50 
51 TOKEN POOLS
52 -----------
53 
54 COBRA `CO_Login_Manager` instances have "token pools." They are supplied (initially by the "God" login, but later, by other `CO_Login_Manager` instances) a "pool" of security tokens.
55 
56 A `CO_Login_Manager` can also create new tokens, which it adds to its pool.
57 
58 When a `CO_Login_Manager` creates another `CO_Login_Manager` or a `CO_Cobra_Login` instance, it can bestow a set of tokens to that new login from its pool. It cannot supply tokens outside of its pool.
59 
60 IMPLEMENTATION
61 ==============
62 
63 You implement COBRA by first setting up \ref CHAMELEON (`CO_Chameleon`), then instantiating `CO_Cobra`, passing in an instance of `CO_Chameleon`, with a `CO_Login_Manager` logged in.
